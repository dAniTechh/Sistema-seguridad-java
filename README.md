Arquitectura del Sistema y Flujo de DatosEste proyecto implementa una arquitectura Full-Stack desacoplada utilizando un modelo de microservicios simplificado. El sistema se divide en un núcleo de procesamiento en Java (Backend) y una interfaz reactiva en React.js (Frontend). La comunicación se realiza mediante una API REST nativa, eliminando la dependencia de frameworks pesados para maximizar el control sobre el ciclo de vida de las peticiones. Todo el ecosistema está orquestado mediante Docker, lo que garantiza la inmutabilidad del entorno de ejecución y facilita el despliegue en cualquier infraestructura.2. Núcleo de Seguridad CriptográficaLa seguridad no se ha tratado como un añadido, sino como el cimiento del sistema. Se ha implementado un esquema de protección de identidad basado en el estándar PBKDF2 (Password-Based Key Derivation Function 2) con HmacSHA256.A diferencia de los hashes simples (como MD5 o SHA-1), este algoritmo aplica 600,000 iteraciones de hashing encadenado, lo que eleva exponencialmente el coste computacional para un atacante. Además, se utiliza una técnica de Salting dinámico mediante la generación de 16 bytes aleatorios por usuario, eliminando la efectividad de los ataques de Rainbow Tables. La gestión de sesiones se realiza mediante Tokens UUID de estado, vinculados a perfiles de usuario protegidos por mecanismos de Anti-Timing y bloqueo temporal tras intentos fallidos.3. Motor de Matching y Algoritmia de MercadoEl componente principal del backend es un Matching Engine de alta concurrencia. El sistema utiliza un algoritmo de Prioridad Precio-Tiempo, el estándar de la industria financiera. Para gestionar las órdenes de compra (Bids) y venta (Asks), se han implementado estructuras de datos de tipo PriorityBlockingQueue.Estas colas operan internamente como un Binary Heap (montículo binario), lo que permite que las operaciones de inserción y extracción de la mejor oferta tengan una complejidad computacional de $O(\log n)$. El motor de cruce garantiza la integridad transaccional: cuando dos órdenes coinciden en precio, el sistema ejecuta un "Match", recalcula los saldos de forma atómica y gestiona ejecuciones parciales (Partial Fills) si las cantidades no coinciden exactamente, reinyectando el remanente en el libro con su prioridad temporal original.4. Gestión de Concurrencia y Persistencia en MemoriaDado que el sistema procesa transacciones financieras en tiempo real, se ha optado por un almacenamiento In-Memory optimizado para hilos (Thread-Safe). Se hace uso intensivo de ConcurrentHashMap para la gestión de saldos y sesiones, permitiendo lecturas concurrentes sin bloqueo y escrituras seguras.La lógica de negocio crítica está protegida mediante Monitores de Sincronización, asegurando que el estado del mercado sea consistente en todo momento. Esto evita errores de doble gasto o condiciones de carrera (Race Conditions) que podrían comprometer la integridad de los datos financieros ante múltiples peticiones simultáneas.5. Integración Full-Stack y DespliegueEl frontend, desarrollado con React y Vite, se comunica con el backend a través de un Proxy Inverso configurado en el servidor de desarrollo. Esta técnica permite eludir las restricciones de política de origen cruzado (CORS) de forma segura durante el desarrollo. El despliegue mediante Docker encapsula la Java Virtual Machine (JVM) y todas las dependencias, permitiendo que el sistema sea agnóstico respecto al sistema operativo host y garantizando que el entorno de desarrollo sea idéntico al de producción.
